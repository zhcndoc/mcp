MCP **`2026-07-28`** 的发布候选版现已可用。这是自发布以来对协议最大的一次
修订，并兑现了
[2026 路线图](/posts/2026-mcp-roadmap/) 中的承诺：

- **一个无状态核心**，可在普通 HTTP 基础设施上扩展
- **扩展**，包括通过 [MCP Apps](#mcp-apps-server-rendered-user-interfaces) 提供的服务端渲染 UI，以及通过 [Tasks 扩展](#tasks-graduates-to-an-extension) 处理长时间运行的工作
- **授权**，与 OAuth 和 OpenID Connect 部署更加一致
- **正式的弃用策略**，使协议能够在不破坏你已构建内容的情况下演进，

以及许多其他变更。

对生产部署的实际影响是立竿见影的。此前需要粘性会话、共享会话存储以及
网关层深度包检测的远程 MCP
服务器，现在可以放在普通的轮询式
负载均衡器之后运行，通过 `Mcp-Method` 标头路由流量，并允许客户端在
服务器的 `ttlMs` 允许的时间内缓存 `tools/list` 响应。

> 发布候选版今天即可使用，最终规范将于
> **2026 年 7 月 28 日** 发布。本次发布包含破坏性变更；详情请参见
> [发布时间线与验证](#release-timeline-and-validation)。

## 无状态协议

最重要的变化是，MCP 现在在协议层面上是无状态的。六项
[规范增强提案](https://modelcontextprotocol.io/community/sep-guidelines)
（SEP）协同工作以实现这一点，完成了我们在
12 月于 [MCP 传输的未来](/posts/2025-12-19-mcp-transport-future/) 中提出的计划。

![之前：客户端通过负载均衡器路由，并粘连到某一个 MCP 服务器实例，所有实例共享一个会话存储。之后：同一个客户端可以被路由到三个 MCP 服务器实例中的任意一个，且不需要会话存储。](/posts/images/2026-07-28-rc/stateless-topology.svg)

### 前后对比

在 [`2025-11-25`](/posts/2025-11-25-first-mcp-anniversary/) 中，通过
Streamable HTTP 调用工具意味着需要先建立会话：

```http
POST /mcp HTTP/1.1
Content-Type: application/json

{"jsonrpc":"2.0","id":1,"method":"initialize",
 "params":{"protocolVersion":"2025-11-25","capabilities":{},
           "clientInfo":{"name":"my-app","version":"1.0"}}}
```

服务器会返回一个 `Mcp-Session-Id`，之后每个请求都必须携带它，
从而将客户端绑定到发出该会话的那个实例：

```http
POST /mcp HTTP/1.1
Mcp-Session-Id: 1868a90c-3a3f-4f5b
Content-Type: application/json

{"jsonrpc":"2.0","id":2,"method":"tools/call",
 "params":{"name":"search","arguments":{"q":"otters"}}}
```

在 `2026-07-28` 中，同样的调用变成了一个任何
服务器实例都可以处理的单个自包含请求：

```http
POST /mcp HTTP/1.1
MCP-Protocol-Version: 2026-07-28
Mcp-Method: tools/call
Mcp-Name: search
Content-Type: application/json

{"jsonrpc":"2.0","id":1,"method":"tools/call",
 "params":{"name":"search","arguments":{"q":"otters"},
           "_meta":{"io.modelcontextprotocol/clientInfo":{"name":"my-app","version":"1.0"}}}}
```

### 握手和会话不再存在

`initialize`/`initialized` 握手已被移除
([SEP-2575](https://github.com/modelcontextprotocol/modelcontextprotocol/pull/2575))。
过去在连接建立时只交换一次的协议版本、客户端信息和客户端能力，
现在会在每个请求的 `_meta` 中传输，而新的 `server/discover` 方法则允许客户端在需要时
提前获取服务器能力。

`Mcp-Session-Id` 标头以及随之而来的协议级会话也已被移除
([SEP-2567](https://github.com/modelcontextprotocol/modelcontextprotocol/pull/2567))。
二者都消失后，任何 MCP 请求都可以落到任意服务器实例上，而此前水平扩展部署所需的粘性
路由和共享会话存储，在协议层面上也不再需要。

### 无状态协议，有状态应用

移除协议层会话并不意味着你的应用必须是
无状态的。需要在多次调用之间保留状态的服务器，可以像 HTTP API 一直以来所做的那样：从某个工具中生成一个显式句柄（例如 `basket_id`、`browser_id`），并让模型在后续调用中将其作为普通参数传回。

![时序图：模型调用 create_basket，MCP 服务器返回一个 basket_id，随后模型在 add_item 中把同一个 basket_id 作为参数传回。](/posts/images/2026-07-28-rc/handle-pattern.svg)

在实践中，我们发现这种模式（让模型把一个标识符从一次工具调用传递到下一次）
不仅仅是会话状态的一个可行替代方案，它往往更强大。模型可以跨工具组合句柄，对其进行推理，
并在步骤之间转交它们，而这些在以前由传输元数据隐藏起来的外部管理会话状态，
实际上从未真正允许这样做。

协议不再替你管理这些状态，但这并不妨碍你自己管理。显式句柄模式只是
让状态对模型可见，而不是把它隐藏起来。

### 服务器到客户端请求的重构

无状态协议仍然需要一种方式，让服务器在调用过程中向客户端请求某些内容，例如
一个 elicitation 提示。两个 SEP 对这一流程进行了重建，使其无需持久连接也能工作。

服务器发起的请求现在只能在服务器正在积极处理客户端请求时发出
([SEP-2260](https://github.com/modelcontextprotocol/modelcontextprotocol/pull/2260))。
更早版本的规范曾建议这样做；现在这已成为强制要求。用户不会莫名其妙地被提示，每一次 elicitation 都能追溯到他们（或他们的代理）发起的某件事。

多轮往返请求
([SEP-2322](https://github.com/modelcontextprotocol/modelcontextprotocol/pull/2322))
改变了这些提示的传递方式。服务器不再保持一个 Server-Sent Events
（SSE）流处于打开状态，而是返回一个 `InputRequiredResult`：

```json
{
  "resultType": "inputRequired",
  "inputRequests": {
    "confirm": {
      "type": "elicitation",
      "message": "Delete 3 files?",
      "schema": { "type": "boolean" }
    }
  },
  "requestState": "eyJzdGVwIjoxLCJmaWxlcyI6WyJhIiwiYiIsImMiXX0="
}
```

客户端收集答案，并带着 `inputResponses` 和回传的 `requestState` 重新发起原始调用。任何服务器实例都可以接手这次重试，因为它所需的一切都已经包含在负载中了。

### 可路由、可缓存、可追踪

三项较小的变更让生成的流量更易于运维。

Streamable HTTP 传输现在要求 `Mcp-Method` 和 `Mcp-Name` 标头
([SEP-2243](https://github.com/modelcontextprotocol/modelcontextprotocol/pull/2243))，
因此负载均衡器、网关和限流器可以基于操作进行路由，而无需检查请求体。
如果标头与请求体不一致，服务器会拒绝请求。

列表和资源读取结果现在携带 `ttlMs` 和 `cacheScope`
([SEP-2549](https://github.com/modelcontextprotocol/modelcontextprotocol/pull/2549))，
其设计类似于 HTTP `Cache-Control`。客户端可以准确知道 `tools/list`
响应何时仍然是新的，以及是否可以安全地跨用户共享，而长时间运行的 SSE
流不再是了解列表变化的唯一方式。

`_meta` 中的 W3C Trace Context 传播现在已被文档化
([SEP-414](https://github.com/modelcontextprotocol/modelcontextprotocol/pull/414))，
从而锁定了 `traceparent`、`tracestate` 和 `baggage` 的键名，使分布式追踪能够在各个 SDK 和网关之间正确关联。许多 SDK 和工具此前就已经这么做了；随着规范中键名的固定，一个从宿主应用开始的追踪现在可以穿过客户端 SDK、MCP 服务器以及服务器下游调用的任何系统，并在一个兼容 [OpenTelemetry](https://opentelemetry.io/) 的后端中显示为单一的 span 树。

## 扩展成为一等公民

扩展在 `2025-11-25` 版本中已经存在，但背后没有正式流程。
[SEP-2133](https://github.com/modelcontextprotocol/modelcontextprotocol/pull/2133)
补上了这一点：扩展通过反向 DNS ID 标识，通过客户端和服务器能力上的 `extensions`
映射进行协商，存放在各自独立的 `ext-*` 仓库中并由委派维护者维护，而且其版本与规范独立演进。SEP 流程中的一个新的 Extensions Track 为它们提供了从实验到正式的路径。

本次发布包含两个官方扩展。

### MCP Apps：服务端渲染用户界面

[MCP Apps](/posts/2026-01-26-mcp-apps/)
([SEP-1865](https://github.com/modelcontextprotocol/modelcontextprotocol/pull/1865))
允许服务器提供交互式 HTML 界面，由宿主在受沙箱保护的 iframe 中渲染。工具会提前声明其 UI 模板，以便宿主在任何内容运行之前就可以预取、缓存并进行安全审查。渲染后的 UI 会通过 MCP 中其他地方使用的同一 JSON-RPC 基础协议与宿主通信，因此每一个由 UI 触发的动作都会与直接工具调用一样，经过相同的审计和同意流程。

### Tasks 升级为扩展

Tasks 在 [`2025-11-25`](https://modelcontextprotocol.io/specification/2025-11-25/server/tasks) 中作为实验性核心功能发布。
生产使用暴露出了足够多需要重设计的问题，因此它更适合放在扩展中，而不是规范本体里。

[Tasks 扩展](https://github.com/modelcontextprotocol/modelcontextprotocol/pull/2663)
围绕无状态模型重塑了生命周期：服务器可以用一个任务句柄响应 `tools/call`，而客户端则通过 `tasks/get`、`tasks/update` 和 `tasks/cancel` 来驱动它。任务创建由服务器指示：客户端声明该扩展，服务器决定何时应将一次调用作为任务运行。`tasks/list` 被移除，因为在没有会话的情况下，它无法安全地进行范围限定。

任何基于 `2025-11-25` 实验性 Tasks API 进行开发的人，都需要迁移到新的生命周期。

## 授权强化

六项 SEP 对
[授权规范](https://modelcontextprotocol.io/specification/draft/basic/authorization)
进行了强化，使其更贴近 OAuth 2.0 和 OpenID Connect 在实践中的部署方式。

客户端现在必须根据
[RFC 9207](https://www.rfc-editor.org/rfc/rfc9207)
在授权响应中验证 `iss` 参数
([SEP-2468](https://github.com/modelcontextprotocol/modelcontextprotocol/pull/2468))。
这是一种低成本缓解措施，可应对一类混淆攻击；在 MCP 的单客户端、多服务器部署模式下，这类攻击更为普遍。在未来版本中，客户端将被要求拒绝省略 `iss` 的响应，因此如果授权服务器尚未提供它，现在就应开始提供。

客户端现在在动态客户端注册期间声明其 OpenID Connect `application_type`
([SEP-837](https://github.com/modelcontextprotocol/modelcontextprotocol/pull/837))，
从而避免授权服务器将桌面或 CLI 客户端默认视为 `"web"` 并拒绝其 localhost 重定向 URI 的常见情况。客户端会将已注册凭据绑定到签发该凭据的授权服务器 `issuer`，并在资源在不同授权服务器之间迁移时重新注册
([SEP-2352](https://github.com/modelcontextprotocol/modelcontextprotocol/pull/2352))。
该规范还记录了如何从 OpenID Connect 风格的授权服务器请求刷新令牌
([SEP-2207](https://github.com/modelcontextprotocol/modelcontextprotocol/pull/2207))，
并阐明了在 step-up 过程中 scope 的累积方式
([SEP-2350](https://github.com/modelcontextprotocol/modelcontextprotocol/pull/2350))
以及 `.well-known` 发现后缀
([SEP-2351](https://github.com/modelcontextprotocol/modelcontextprotocol/pull/2351))。

## Roots、Sampling 和 Logging 已弃用

在新的
[功能生命周期策略](#how-the-protocol-evolves-from-here)
([SEP-2577](https://github.com/modelcontextprotocol/modelcontextprotocol/pull/2577)) 下，三个核心功能已弃用：

| 功能      | 替代方案                                                                                         |
| --------- | ------------------------------------------------------------------------------------------------ |
| Roots     | [工具参数](https://modelcontextprotocol.io/specification/draft/server/tools)、资源 URI 或服务器配置 |
| Sampling  | 与 LLM 提供商 API 直接集成                                                                       |
| Logging   | 对于 stdio 传输使用 `stderr`；对于结构化可观测性使用 [OpenTelemetry](https://opentelemetry.io/) |

这些都是仅标注式弃用。这些方法、类型和能力标志在本次发布以及其后一年的每个规范版本中仍将继续工作，而移除其中任何一项都需要根据生命周期策略另行提交 SEP。

## 工具的完整 JSON Schema 2020-12

工具的 `inputSchema` 和 `outputSchema` 已提升为完整的
[JSON Schema 2020-12](https://json-schema.org/draft/2020-12)
([SEP-2106](https://github.com/modelcontextprotocol/modelcontextprotocol/pull/2106))。
输入 schema 仍保留 `type: "object"` 的根约束，但现在允许组合（`oneOf`、`anyOf`、`allOf`）、条件以及引用（`$ref`、`$defs`）。输出 schema 不再受限制，且 `structuredContent` 现在可以是任意 JSON 值，而不再仅限于对象。实现不得自动解除外部 `$ref` URI 的引用，并且应限制 schema 深度和验证时间。

另外，缺少资源时的错误码从 MCP 自定义的 `-32002` 改为 JSON-RPC 标准的 `-32602` Invalid Params
([SEP-2164](https://github.com/modelcontextprotocol/modelcontextprotocol/pull/2164))。
如果你的客户端是按字面值 `-32002` 来匹配的，请将其更新。

## 协议将如何从这里演进

本次发布包含破坏性变更。我们并不希望这成为常态。

本次发布中的三项治理 SEP 旨在使未来修订能够在不破坏核心能力的前提下演进协议。  
[功能生命周期策略](https://github.com/modelcontextprotocol/modelcontextprotocol/pull/2596)
为每个功能提供 _Active_、_Deprecated_ 和 _Removed_ 生命周期，并且从弃用到最早可能移除之间至少间隔十二个月。  
[扩展框架](#extensions-become-first-class) 意味着新能力可以作为可选扩展发布，并在此处稳定下来，然后再——如果有的话——进入规范。  
此外，标准轨道 SEP 在匹配场景落入 [conformance suite](https://github.com/modelcontextprotocol/conformance)
([SEP-2484](https://github.com/modelcontextprotocol/modelcontextprotocol/pull/2484)) 之前，不能再达到 Final 状态，
而新的 [SDK 分级系统](https://github.com/modelcontextprotocol/modelcontextprotocol/pull/1777)
正是用同一套测试套件来评估官方 SDK 的。

本次发布中的无状态重构属于那种需要干净切换的基础性变更。随着它的落地，以及随着弃用窗口和扩展成为未来的标准工具，我们预计面向 `2026-07-28` 的实现者将能够采用未来修订，而无需重写其传输或生命周期代码。

## 发布时间线与验证

候选发布版已于 **2026 年 5 月 21 日** 锁定。最终规范将于 **2026 年 7 月 28 日** 发布。这十周窗口供 SDK 维护者和客户端实现者结合真实工作负载验证这些变更；在 [SDK 分级系统](https://modelcontextprotocol.io/docs/sdk) 下，Tier 1 SDK 预计应在此窗口内提供支持。

完整候选发布版位于
[草案规范](https://modelcontextprotocol.io/specification/draft) 中，而
[changelog](https://modelcontextprotocol.io/specification/draft/changelog)
将列出相对于 `2025-11-25` 的每一项变更。

如果你发现问题，请在
[规范仓库](https://github.com/modelcontextprotocol/modelcontextprotocol/issues) 中提交 issue。
对于实现相关问题，最便捷的答疑路径是
[贡献者 Discord](https://modelcontextprotocol.io/community/communication#discord)
中的相应
[工作组](https://modelcontextprotocol.io/community/working-interest-groups)
频道。

## 展望未来

本次发布为 MCP 奠定了我们预计将长期建立其上的基础：一个可在通用 HTTP 基础设施上无状态运行的协议，一个扩展框架，使 Tasks 和 MCP Apps 之类的能力能够按各自时间线发布，以及一项生命周期策略，让实现者能够基于 `2026-07-28` 构建，同时确信他们交付的内容会持续工作。

感谢所有通过各个工作组以及大量耐心审阅为这些提案做出贡献的人。我们期待在 7 月 28 日与社区一起将其最终定稿。
