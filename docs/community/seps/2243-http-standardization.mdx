## 摘要

本 SEP 提议在 Streamable HTTP 传输的标准 HTTP 头部位置公开关键的路由与上下文信息。通过将 JSON-RPC 载荷中的关键字段镜像到 HTTP 头部，负载均衡器、代理和可观测性工具等网络中介可以在无需深度包检测的情况下对 MCP 流量进行路由和处理，从而降低延迟与计算开销。

## 动机

当前通过 HTTP 实现的 MCP 会将所有路由信息都埋在 JSON-RPC 载荷中，这会给网络基础设施带来摩擦：

- **负载均衡器** 必须终止 TLS 并解析整个 JSON 正文以提取路由信息（例如区域、工具名称）
- **代理和网关** 无法在没有深度包检测的情况下做出路由决策
- **可观测性工具** 对 MCP 流量模式的可见性有限
- **限流器和 WAF** 无法基于 MCP 特定字段应用策略

通过在 HTTP 头中公开关键字段，我们使标准网络基础设施能够使用现有且广泛支持的机制处理 MCP 流量。

## 规范

### 标准头部

Streamable HTTP 传输将要求 POST 请求包含以下从请求体镜像而来的头部：

| Header Name  | Source Field                  | Required For                                           |
| ------------ | ----------------------------- | ------------------------------------------------------ |
| `Mcp-Method` | `method`                      | 所有请求                                               |
| `Mcp-Name`   | `params.name` or `params.uri` | `tools/call`、`resources/read`、`prompts/get` 请求     |

这些头部在引入它们的 MCP 版本中是**必需**的。

**服务器行为**：处理请求体的服务器 MUST 拒绝那些头部中指定值与请求体中的值不匹配的请求。

> **理由**：这一要求可防止网络中不同组件依赖不同“事实来源”而可能产生的安全漏洞和错误条件。例如，负载均衡器或网关可能使用头部值做路由决策，而 MCP 服务器使用正文值执行操作。该要求适用于任何处理消息体的网络中介，以及 MCP 服务器本身。

**大小写敏感性**：头名称（在 [RFC 9110](https://datatracker.ietf.org/doc/html/rfc9110#name-field-names) 中称为“字段名”）是不区分大小写的。客户端和服务器 MUST 对头名称使用不区分大小写的比较。

### 来自工具参数的自定义头部

MCP 服务器 MAY 使用工具的 `inputSchema` 中参数 schema 里的 `x-mcp-header` 扩展属性，指定某些工具参数应镜像到 HTTP 头部中。

**客户端要求**：尽管服务器使用 `x-mcp-header` 是可选的，但客户端 MUST 支持此功能。当服务器的工具定义包含 `x-mcp-header` 注解时，符合规范的客户端 MUST 按本文档规定将指定的参数值镜像到 HTTP 头中。

#### Schema 扩展

`x-mcp-header` 属性指定用于构造头名称 `Mcp-Param-{name}` 的名称部分。

**`x-mcp-header` 值的约束**：

- MUST NOT 为空
- MUST 仅包含 ASCII 字符（不包括空格和 `:`）
- 在 `inputSchema` 中所有 `x-mcp-header` 值之间 MUST 具有不区分大小写的唯一性
- MUST 只应用于原始类型参数（number、string、boolean）

客户端 MUST 拒绝任何 `x-mcp-header` 值违反这些约束的工具定义。拒绝的含义是客户端 MUST 将无效工具从 `tools/list` 的结果中排除。客户端 SHOULD 在拒绝工具定义时记录警告，包括工具名称和拒绝原因。该行为可确保单个格式错误的工具定义不会阻止其他有效工具的使用。

**工具定义示例**：

```json
{
  "name": "execute_sql",
  "description": "在 Google Cloud Spanner 上执行 SQL",
  "inputSchema": {
    "type": "object",
    "properties": {
      "region": {
        "type": "string",
        "description": "执行查询所在的区域",
        "x-mcp-header": "Region"
      },
      "query": {
        "type": "string",
        "description": "要执行的 SQL 查询"
      }
    },
    "required": ["region", "query"]
  }
}
```

#### 示例：地理分布式数据库

考虑一个为 Google Cloud Spanner 暴露 `execute_sql` 工具的服务器，该工具需要一个 `region` 参数。

**工具定义**：

```json
{
  "name": "execute_sql",
  "description": "在 Google Cloud Spanner 上执行 SQL",
  "inputSchema": {
    "type": "object",
    "properties": {
      "region": {
        "type": "string",
        "description": "执行查询所在的区域",
        "x-mcp-header": "Region"
      },
      "query": {
        "type": "string",
        "description": "要执行的 SQL 查询"
      }
    },
    "required": ["region", "query"]
  }
}
```

**场景**：客户端请求在 `us-west1` 中执行 SQL。

**当前摩擦**：全局负载均衡器收到了请求，但它必须终止 TLS 并解析整个 JSON 正文，找到 `"region": "us-west1"`，然后才能知道应将数据包路由到俄勒冈还是比利时集群。

**请求**：

```http
POST /mcp HTTP/1.1

Mcp-Session-Id: 1f3a4b5c-6d7e-8f9a-0b1c-2d3e4f5a6b7c
Mcp-Method: tools/call
Mcp-Name: execute_sql
Mcp-Param-Region: us-west1

{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "tools/call",
  "params": {
    "name": "execute_sql",
    "arguments": {
      "region": "us-west1",
      "query": "SELECT * FROM users"
    }
  }
}
```

### 头部处理

#### 值编码

客户端 MUST 在将参数值包含到 HTTP 头中之前对其进行编码，以确保安全传输并防止注入攻击。

**字符限制**

根据 [RFC 9110](https://datatracker.ietf.org/doc/html/rfc9110#name-field-values)，HTTP 头字段值必须由可见 ASCII 字符（0x21-0x7E）、空格（0x20）和水平制表符（0x09）组成。以下字符被明确禁止：

- 回车（`\r`，0x0D）
- 换行（`\n`，0x0A）
- 空字符（`\0`，0x00）
- ASCII 范围之外的任何字符（> 0x7F）

**空白处理**

HTTP 解析器通常会去除头值前后的空白。为了保留参数值中的前导和尾随空格，客户端 MUST 在值满足以下条件时使用 Base64 编码：

- 以空格（0x20）或水平制表符（0x09）开头
- 以空格（0x20）或水平制表符（0x09）结尾

**编码规则**

客户端 MUST 按以下顺序应用编码规则：

1. **类型转换**：将参数值转换为其字符串表示形式：
   - `string`：原样使用值
   - `number`：转换为十进制字符串表示（例如 `42`、`3.14`）
   - `boolean`：转换为小写 `"true"` 或 `"false"`

2. **空白检查**：如果字符串以空白（空格或制表符）开头或结尾：
   - 应用 Base64 编码（见下文）

3. **ASCII 验证**：检查字符串是否只包含有效的 ASCII 字符（0x20-0x7E）：
   - 如果有效，继续执行第 4 步
   - 如果无效（包含非 ASCII 字符），应用 Base64 编码（见下文）

4. **控制字符检查**：如果字符串包含任何控制字符（0x00-0x1F 或 0x7F）：
   - 应用 Base64 编码（见下文）

**不安全值的 Base64 编码**

当某个值无法安全地表示为普通 ASCII 头值时，客户端 MUST 使用该值的 UTF-8 表示进行 Base64 编码，格式如下：

```text
Mcp-Param-{Name}: =?base64?{Base64EncodedValue}?=
```

前缀 `=?base64?` 和后缀 `?=` 表示该值已进行 Base64 编码。需要检查这些值的服务器和中介 MUST 相应地解码。

**示例**：

| Original Value   | Reason                  | Encoded Header Value                                  |
| ---------------- | ----------------------- | ----------------------------------------------------- |
| `"us-west1"`     | 纯 ASCII                | `Mcp-Param-Region: us-west1`                          |
| `"Hello, 世界"`  | 包含非 ASCII             | `Mcp-Param-Greeting: =?base64?SGVsbG8sIOS4lueVjA==?=` |
| `" padded "`     | 前导/尾随空格           | `Mcp-Param-Text: =?base64?IHBhZGRlZCA=?=`             |
| `"line1\nline2"` | 包含换行                 | `Mcp-Param-Text: =?base64?bGluZTEKbGluZTI=?=`         |

#### 客户端行为

当通过 HTTP 传输构造 `tools/call` 请求时，客户端 MUST：

1. 从请求体中提取任何标准头对应的值（例如 `method`、`params.name`、`params.uri`）
1. 将 `Mcp-Method` 头以及（如适用）`Mcp-Name` 头附加到请求中
1. 检查工具的 `inputSchema` 中带有 `x-mcp-header` 标记的属性，并提取每个参数的值
1. 按照 [值编码](#值编码) 中的规则对这些值进行编码
1. 将 `Mcp-Param-{Name}: {Value}` 头附加到请求中：

#### 服务器行为

接收到请求时，服务器 MUST 拒绝包含无效字符的 `Mcp-Param-{Name}` 头（参见 [值编码](#值编码) 部分中的“字符限制”）。

任何处理消息体（而非仅仅转发消息体）的服务器 MUST 验证：若头值经过 Base64 编码，则在解码后，这些编码后的头值与请求体中的对应值相匹配。若任何验证失败，服务器 MUST 以 `400 Bad Request` HTTP 状态拒绝请求。

**错误代码**

当由于头验证失败而拒绝请求时，服务器 MUST 返回带有以下错误代码的 JSON-RPC 错误响应：

| Code     | Name             | Description                                                                                                            |
| -------- | ---------------- | ---------------------------------------------------------------------------------------------------------------------- |
| `-32001` | `HeaderMismatch` | HTTP 头与请求体中的对应值不匹配，或必需头缺失/格式错误。                                                             |

此错误代码位于 JSON-RPC 实现定义的服务器错误范围（`-32000` 到 `-32099`）。

**错误响应格式**：

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "error": {
    "code": -32001,
    "message": "头不匹配：Mcp-Name 头值 'foo' 与正文值 'bar' 不匹配"
  }
}
```

**验证失败条件**：

- 缺少必需的标准头（`Mcp-Method`、`Mcp-Name` 等）
- 头值与请求体值不匹配
- 无法解码 Base64 编码的值
- 头值包含无效字符

> **注意**：中介 MUST 在验证失败时返回适当的 HTTP 错误状态（例如 `400 Bad Request`），但不要求返回 JSON-RPC 错误响应。

**自定义头处理**：

通过 `x-mcp-header` 定义的自定义头遵循与标准头相同的验证规则：

| Scenario                                 | Client Behavior                | Server Behavior                          |
| ---------------------------------------- | ------------------------------ | ---------------------------------------- |
| Parameter value provided                 | Client MUST include the header | Server MUST validate header matches body |
| Parameter value is `null`                | Client MUST omit the header    | Server MUST NOT expect the header        |
| Parameter not in arguments               | Client MUST omit the header    | Server MUST NOT expect the header        |
| Client omits header but value is in body | Non-conforming client          | Server MUST reject the request           |

当由于缺少或无效的自定义头而拒绝请求时，服务器 MUST 返回 HTTP 状态 `400 Bad Request`，并附带 JSON-RPC 错误代码 `-32001`（`HeaderMismatch`）。

## 原理说明

### Headers vs Path

本提案将请求数据镜像到 headers 中，而不是将其编码到 URL path 中。

**Headers 的优势**：

1. **简单性**：所有广泛使用的网络负载均衡器都支持基于 HTTP headers 的路由
2. **多版本支持**：在客户端和服务器中支持多个 MCP 版本更容易
3. **兼容性**：Headers 可与现有的 Streamable HTTP 传输设计配合使用，而无需更改 endpoint 结构
4. **值无限制**：Header 值可以包含在 URL 中需要编码的字符（例如 `/`、`?`、`#`）
5. **无 URL 长度限制**：可以传输非常长的值，而不会触及 URL 长度限制

**基于 Path 路由的优势**：

1. **框架简单性**：许多 Web 框架（Flask、Express、Django、Rails）内置了对基于 path 路由的支持，只需很少的配置
2. **日志记录**：URL paths 通常默认会被记录下来，因此更容易调试

**权衡与框架考量**：

| Framework         | 基于 Header 的路由                                             | 基于 Path 的路由                           |
| ----------------- | --------------------------------------------------------------- | ------------------------------------------ |
| Flask (Python)    | 需要中间件或装饰器在路由前提取 headers | 原生支持 `@app.route('/mcp/<method>')` |
| Express (Node.js) | 通过 `req.headers` 很容易实现，但需要自定义路由逻辑            | 原生支持 `app.post('/mcp/:method')`    |
| Django (Python)   | 需要自定义中间件                                          | 原生 URL patterns                              |
| Go (net/http)     | 通过 `r.Header.Get()` 很容易实现                                           | 通过 path patterns 原生支持                         |
| ASP.NET Core      | 通过 `[FromHeader]` attribute 很容易实现                                   | 通过 route templates 原生支持                       |

对于像 Flask 这样强烈偏好基于 path 路由的框架，实现基于 header 的路由需要额外代码：

```python
# Flask 示例：基于 Header 的路由需要手动分发
@app.route('/mcp', methods=['POST'])
def mcp_handler():
    method = request.headers.get('Mcp-Method')
    if method == 'tools/call':
        return handle_tools_call(request)
    elif method == 'resources/read':
        return handle_resources_read(request)
    # ... 等等
```

尽管在某些框架中会增加这些额外复杂性，但最终仍选择基于 header 的路由，因为：

1. **向后兼容性**：引入基于 path 的路由会要求所有现有 MCP Servers 进行重大更新，并且可能需要同时支持两套 endpoints 以支持多个版本。即使 SDK 能部分掩盖这些问题，诸如测试、metrics 等额外运维工作仍然需要处理。基于 header 的路由对客户端侧的改动最小。未选择 opt in 的客户端也仍然可以正常工作。

2. **基础设施收益大于框架复杂性**：主要目标是让网络基础设施（负载均衡器、代理、WAF）无需解析 body 就能对请求进行路由和处理。无论服务器使用何种框架，这一收益都成立。

### 基础设施支持

HTTP 基于 header 的路由和处理受到以下组件支持：

- **Load Balancers**：所有主流负载均衡器（HAProxy、NGINX、Cloudflare、F5、Envoy/Istio）
- **Rate Limiting**：11 个流行限流方案中的 9 个
- **Authorization**：Kong、Tyk、AWS API Gateway、Google Cloud Apigee、Azure API Gateway、NGINX、Apache APISIX、Istio/Envoy
- **Web Application Firewalls**：Cloudflare WAF、AWS WAF、Azure WAF、F5 Advanced WAF、FortiWeb、Imperva WAF、Barracuda WAF、ModSecurity、Akamai、Wallarm
- **Observability**：大多数可观测性方案都可以从 HTTP headers 中提取数据

### x-mcp-header 中的显式 Header 名称

该设计在 `x-mcp-header` 中使用显式的名称值，而不是从参数名推导 header 名称，原因如下：

1. **大小写敏感性不匹配**：Header 名称不区分大小写，但 JSON Schema 属性名区分大小写
2. **字符集约束**：Header 名称仅限于 ASCII 字符，而工具参数名可能包含任意 Unicode
3. **简单性**：不需要从嵌套属性构造 header 名称的复杂方案

### 在 JSON Schema 中的位置

`x-mcp-header` 扩展直接放置在要镜像的属性的 JSON Schema 内部，而不是放在 schema 之外的单独元数据字段中。该设计选择具有以下优势：

1. **共置**：header 映射与其影响的属性定义在一起，使其立即清楚哪个参数会被镜像。开发者无需在 schema 与单独的元数据结构之间来回查找。

2. **成熟模式**：JSON Schema 明确支持扩展关键字（以 `x-` 开头的属性），而且这种模式在 OpenAPI 等生态中被广泛使用。工具作者和 SDK 开发者已经对这种做法很熟悉。

3. **Schema 可组合性**：当 schemas 被组合、扩展或通过 `$ref` 引用时，`x-mcp-header` 注解会跟随属性定义一起传播。单独的元数据结构则需要复杂的同步逻辑来保持一致性。

4. **工具兼容性**：现有的 JSON Schema 验证器默认会忽略未知关键字，因此添加 `x-mcp-header` 不会破坏现有的 schema 验证。不理解此扩展的工具只会直接跳过它。

5. **降低复杂性**：单独的元数据结构需要定义一种映射机制（例如 JSON Pointer 或 property paths）来将 headers 与属性关联起来，从而增加实现复杂度和出错可能性。

### 范围：仅限 Tools

`x-mcp-header` 机制目前仅适用于 `tools/call` 请求，因为 tools 是唯一具有 `inputSchema`、并支持 JSON Schema 扩展关键字的 MCP 原语。Resources 和 prompts 缺少等价的 schema 结构：`resources/read` 只接受一个 `uri`（已通过 `Mcp-Name` 暴露），而 `prompts/get` 将参数定义为一个简单的 `{name, description, required}` 数组，不具备 JSON Schema 扩展性。将自定义 header 映射泛化到这些原语，需要为 resources 和 prompts 增加类似 `inputSchema` 的定义，这将是更大的规范变更。这里将其标记为未来可能的扩展。

### 不定义规范层面的 Header 大小限制

本规范有意不定义单个 header 值长度、MCP header 总大小或自定义 header 数量的限制。Headers 本质上是 HTTP 概念，而 HTTP 本身（[RFC 9110](https://datatracker.ietf.org/doc/html/rfc9110)）并未规定 header 大小限制。常见的 HTTP 基础设施会施加自己的限制——某些服务器大约为 4–8 KB（例如 Apache 约 8190 bytes），而其他一些则可达到 128 KB（例如 Cloudflare）——但合适的限制取决于部署环境，而这只有服务运营者能够确定。

定义一个规范层面的限制（例如“省略超过 8192 bytes 的 headers”）会带来问题：

1. **任意阈值**：任何选定的值对某些部署来说都太低，而对另一些部署则无关紧要。正确的限制取决于基础设施。
2. **反效果的省略**：如果客户端因为 header 超过规范定义的限制而省略它，那么依赖该 header 进行路由的服务器和中间件就必须解析 body 或拒绝请求——这削弱了将值暴露在 headers 中的核心目的。
3. **不必要的 SDK 负担**：SDK 维护者需要为一个在实践中很少适用的约束实现并测试限制检查逻辑。
4. **与 HTTP 重复**：服务器和中间件已经会使用标准 HTTP 状态码拒绝过大的 headers（`413 Request Entity Too Large`、`431 Request Header Fields Too Large`），无论如何客户端都必须处理这些情况。

> **给实现者的说明**：服务器、中间件和客户端可以根据其部署环境独立地对单个 header 大小、MCP header 总大小或自定义 header 数量施加限制。服务器应当记录其施加的任何限制。客户端应优雅地处理 `413 Request Entity Too Large` 或 `431 Request Header Fields Too Large` 响应。工具作者应将 `x-mcp-header` 注解限制在能带来明确基础设施收益的参数上。

### 不安全值的编码方法

对于无法安全表示为普通 ASCII header 值的参数值（非 ASCII 字符、前导/尾随空白、控制字符），考虑了四种编码方法：

1. **Sentinel wrapping（已选方案）**：在同一个 `Mcp-Param-{Name}` header 中使用 `=?base64?{value}?=` 前缀/后缀来表示值已进行 Base64 编码。

2. **独立 header 名称**：为编码值使用不同的 header 名称，例如 `Mcp-ParamEncoded-{Name}`，通过 header 名称而不是值格式来表示编码方式。

3. **隐式编码**：让解析器根据工具 schema 推断编码，例如在工具定义中通过 `"x-mcp-header-encoding": "base64"` 注解。

4. **始终编码**：无条件对每个 `Mcp-Param-{Name}` 值进行 Base64 编码。

| Approach             | 优点                                                                                                                                     | 缺点                                                                                                                                                                                          |
| -------------------- | ---------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Sentinel wrapping    | 每个参数只需一个 header 名称；常见情况（普通 ASCII）可直接阅读；中间件无需解码即可基于普通值进行路由 | 带内信号在理论上可能与字面值冲突；每个读取方都必须检查前缀                                                                                       |
| Separate header name | 无带内歧义；编码方式可从 header 名称自说明                                                                  | header 命名空间翻倍；每个中间件必须为每个参数检查两个 header 名称；如果两者同时存在，需要一个冲突规则                                                         |
| Implicit encoding    | 线上的格式最简单；没有 sentinel 或额外 headers                                                                                      | 中间件需要访问工具 schema 才知道是否要解码——这违背了将值暴露在 headers 中的目的；按参数静态决定无法很好处理混合情况 |
| Always encode        | 规则最简单；没有条件逻辑或歧义                                                                                        | 普通 ASCII 值会变得不可读；中间件必须解码 Base64 才能检查任何值，显著削弱了本 SEP 的核心动机                                       |

**结论**：Sentinel wrapping 方法提供了最佳权衡。自定义 headers 的主要用途是让中间件能够基于诸如 region 名称和 tenant ID 之类简单、可读的值进行路由和过滤——这些值几乎总是纯 ASCII，也永远不会触发 Base64 编码。选项 4 会让所有值对中间件都变成不可见。选项 3 会让中间件在没有工具 schema 的情况下无法区分编码值与字面值。选项 2 消除了带内歧义，但使 header 命名空间翻倍，要求中间件为每个参数检查两种可能的 header 名称，并在两者同时存在时增加冲突规则。选项 1 中 sentinel 的理论碰撞风险可以忽略不计，因为 `=?base64?...?=` 在实践中作为字面参数值出现的可能性极低。

## 向后兼容性

### 标准头

在使用新的 MCP 版本时，现有客户端和 SDK 将被要求包含标准头。这只是一个小的新增，因为客户端已经包含了类似 `Mcp-Protocol-Version` 的头，每条消息只需再增加一两个新头。

实现新版本的服务器 MUST 拒绝使用新版本或更高版本但缺少必需头的请求。服务器 MAY 通过在协商较旧协议版本时接受不带头的请求来支持旧客户端。

只有在存在 `Mcp-Protool-Version` 头并且其值设置为新协议版本或更高时，中间服务器才能依赖头值。

### 来自工具参数的自定义头

`x-mcp-header` 扩展对服务器来说是可选的。没有此属性的现有工具将继续按原样工作。不过，实现包含本规范的 MCP 版本的客户端 MUST 支持该特性。不支持 `x-mcp-header` 的旧客户端仍然可以正常工作，但不会提供服务器可能依赖的基于头的路由优势。

## 安全影响

### 头注入

当包含控制字符（尤其是 `\r\n`）的恶意值被包含在头中时，就会发生头注入攻击，这可能允许攻击者注入额外的头或提前终止头部部分。

客户端 MUST 遵循本规范中定义的 [值编码](#value-encoding) 规则。这些规则确保：

- 控制字符绝不会出现在头值中
- 非 ASCII 值会使用 Base64 安全编码
- 超出安全长度限制的值会被省略

### 头伪造

服务器 MUST 验证头值与请求体中的对应值一致。这可防止客户端发送不匹配的头以操纵路由，同时执行不同的操作。

例如，恶意客户端可能尝试：

- 将请求路由到安全性较低的区域，同时执行原本 intended for a high-security region 的操作
- 通过伪造租户标识符绕过速率限制
- 通过虚假表示正在执行的操作来规避安全策略

### 信息泄露

被指定用于头的工具参数值对网络中间方（负载均衡器、代理、日志系统）可见。服务器开发者：

- SHOULD NOT 使用 `x-mcp-header` 标记敏感参数（密码、API 密钥、令牌、PII）
- SHOULD 记录哪些参数会作为头暴露
- SHOULD 考虑 Base64 编码并不提供保密性——它只是编码，不是加密

### 信任头值

头值来源于工具调用参数，而这些参数可能受到 LLM 或恶意客户端的影响。中间方和服务器 MUST NOT 将这些值视为用于安全敏感决策的可信输入。特别是：

- 表示对特定资源访问权限的头值（例如租户 ID、区域名称）在授予访问这些资源之前 MUST 依据经过认证用户的权限进行独立验证。
- 在没有服务器端强制执行速率限制和配额的情况下，头值 MUST NOT 作为授予更高权限的唯一依据。
- 部署 SHOULD 尽早在流水线中拒绝带有过大或过多头的请求——在执行 Base64 解码或正文解析之前——以减轻来自构造负载的拒绝服务风险。
