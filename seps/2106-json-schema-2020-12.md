## 摘要

本 SEP 提议放宽对 `inputSchema`、`outputSchema` 和 `structuredContent` 的限制，以更好地支持 JSON Schema 2020-12。具体而言：

- **`inputSchema`**：保留必需的 `type: "object"`（因为工具参数是对象），但允许任何额外的 JSON Schema 属性，以支持强大的验证组合（`anyOf`、`oneOf`、`allOf` 等）
- **`outputSchema`**：由于 MCP 服务器可能返回任何有效的 JSON，因此完全支持 JSON Schema 2020-12
- **`structuredContent`**：接受由 `outputSchema` 验证通过的任意 JSON 值

本提案使 MCP 服务器能够利用 JSON Schema 2020-12 的表达能力，同时保持与现有实现的向后兼容。

## 动机

当前的 MCP 规范以一些方式限制了工具 schema，这与完整的 JSON Schema 支持相冲突：

1. **inputSchema 限制**：目前只允许 `type`、`properties` 和 `required` 字段。这阻止了使用 `anyOf`、`oneOf` 和 `allOf` 等组合关键字来实现复杂的对象验证模式。

2. **outputSchema 限制**：同样仅限于 `type: "object"`，以及 `properties` 和 `required`，尽管规范声称支持“JSON Schema”。

3. **structuredContent 限制**：定义为 `{ [key: string]: unknown }`（一个以字符串为键的对象），这阻止了返回数组——而数组是一种常见的 API 响应模式。

### 实际影响

考虑一个返回逐小时预报的天气 API 工具：

```json
[
  { "hour": "09:00", "temp": 68, "conditions": "sunny" },
  { "hour": "10:00", "temp": 72, "conditions": "partly cloudy" },
  { "hour": "11:00", "temp": 75, "conditions": "cloudy" }
]
```

目前，这种自然的数组响应是**不可能的**，因为 `structuredContent` 必须是对象。开发者被迫将数组包裹在多余的容器对象中：

```json
{
  "forecasts": [
    { "hour": "09:00", "temp": 68, "conditions": "sunny" },
    ...
  ]
}
```

这种人为约束：

- 为响应增加了不必要的嵌套
- 与常见的 REST API 模式相冲突
- 阻止了对数组响应进行直接的 schema 验证

### Schema 组合使用场景

当前对 `inputSchema` 的限制阻止了合理的 schema 模式。借助本 SEP，工具可以在 `type: "object"` 之外使用组合关键字：

```json
{
  "type": "object",
  "oneOf": [
    { "properties": { "id": { "type": "string" } }, "required": ["id"] },
    { "properties": { "name": { "type": "string" } }, "required": ["name"] }
  ]
}
```

这种模式允许工具接受基于 ID 或基于名称的查询——这是一种常见的 API 设计，但目前不受支持，因为该 schema 只允许 `type`、`properties` 和 `required` 字段。

## 规范

### 1. 放宽 inputSchema

**当前定义：**

```typescript
inputSchema: {
  type: "object";
  properties?: { [key: string]: object };
  required?: string[];
};
```

**提议定义：**

```typescript
inputSchema: {
  $schema?: string;
  type: "object";
  [key: string]: unknown;
};
```

`inputSchema` 字段保留 `type: "object"` 的要求（因为工具参数始终是对象），但现在接受任何额外的 JSON Schema 属性。这使得可以支持：

- 组合关键字：`anyOf`、`oneOf`、`allOf`、`not`
- 条件 schema：`if`/`then`/`else`
- 引用 schema：`$ref`、`$defs`
- 任何其他有效的 JSON Schema 2020-12 关键字

### 2. 放宽 outputSchema

**当前定义：**

```typescript
outputSchema?: {
  type: "object";
  properties?: { [key: string]: object };
  required?: string[];
};
```

**提议定义：**

```typescript
outputSchema?: {
  $schema?: string;
  [key: string]: unknown;
};
```

`outputSchema` 字段接受任何有效的 JSON Schema 2020-12 对象，从而可以定义验证数组、基本类型或复杂组合的 schema。与 `inputSchema` 不同，这里没有 `type: "object"` 的要求，因为工具输出可以是任何有效的 JSON。

### 3. 放宽 structuredContent

**当前定义：**

```typescript
structuredContent?: { [key: string]: unknown };
```

**提议定义：**

```typescript
structuredContent?: unknown;
```

`structuredContent` 字段接受任何符合工具 `outputSchema` 的有效 JSON 值。这包括：

- 对象：`{ "key": "value" }`
- 数组：`[1, 2, 3]` 或 `[{ "id": "abc" }, { "id": "xyz" }]`
- 基本类型：`"string"`、`42`、`true`、`null`

### 4. 文档更新

更新 `docs/specification/draft/server/tools.mdx`：

- 删除 `structuredContent` “以 JSON 对象返回”的说明
- 明确 `structuredContent` 可以是任何符合 `outputSchema` 的 JSON 值
- 添加演示数组响应的示例

### 5. 示例

#### 返回对象数组的工具：

```json
{
  "name": "list_users",
  "description": "List all users in the system",
  "inputSchema": {
    "type": "object",
    "properties": {
      "limit": { "type": "integer", "minimum": 1, "maximum": 100 }
    }
  },
  "outputSchema": {
    "type": "array",
    "items": {
      "type": "object",
      "properties": {
        "id": { "type": "string" },
        "name": { "type": "string" },
        "email": { "type": "string", "format": "email" }
      },
      "required": ["id", "name"]
    }
  }
}
```

响应：

```json
{
  "content": [
    {
      "type": "text",
      "text": "Found 2 users: Alice (u1, alice@example.com) and Bob (u2, bob@example.com)."
    }
  ],
  "structuredContent": [
    { "id": "u1", "name": "Alice", "email": "alice@example.com" },
    { "id": "u2", "name": "Bob", "email": "bob@example.com" }
  ]
}
```

#### 使用组合 schema 的工具：

```json
{
  "name": "find_resource",
  "description": "Find a resource by ID or name",
  "inputSchema": {
    "type": "object",
    "oneOf": [
      {
        "properties": { "id": { "type": "string", "format": "uuid" } },
        "required": ["id"]
      },
      {
        "properties": { "name": { "type": "string", "minLength": 1 } },
        "required": ["name"]
      }
    ]
  }
}
```

## 理由

### 为什么不只是允许数组？

虽然我们可以简单地将 `structuredContent` 扩展为允许数组，但这会是一个不完整的解决方案。根本原因在于 schema 类型被人为限制为 `type: "object"`。通过允许任何有效的 JSON Schema，我们可以：

1. 充分发挥 JSON Schema 2020-12 的全部能力
2. 与规范声称支持 JSON Schema 保持一致
3. 提供一致、原则性的解决方案，而不是零敲碎打的修补

### 为什么不要求包装对象？

要求将数组包裹在对象中（例如 `{ "items": [...] }`）曾被考虑，但最终被拒绝，因为：

1. 它会为响应增加不必要的复杂性
2. 它与常见的 API 设计模式相冲突
3. 它阻止了对实际响应结构进行直接的 schema 验证
4. JSON Schema 已经能够优雅地处理数组验证

### 真实世界的 API 模式

许多生产环境 API 会直接返回数组：

- **GitHub Events API**：返回事件对象数组
- **AccuWeather Search API**：返回位置匹配结果数组
- **REST 集合端点**：标准的 `GET /users` 返回 `[{...}, {...}]`

强制使用包装对象会给将现有 API 集成到 MCP 的开发者带来摩擦。通用的 JSON Schema 验证库应该无需 MCP 特定定制即可工作。

### 与 JSON Schema 2020-12 的一致性

JSON Schema 2020-12 为 schema 组合和验证提供了强大的特性。通过移除人为限制，MCP 与行业标准保持一致（OpenAPI 3.1 使用 JSON Schema 2020-12），并使开发者能够利用已有的 JSON Schema 知识和工具链。

### SDK 生态证据

当前限制所造成的摩擦并非理论上的。FastMCP 作为最流行的 MCP Python SDK 之一，已经实现了大量变通方案：

1. **显式错误消息** 直接说明了这一限制：

   ```python
   raise ValueError(
       f"Output schemas must represent object types due to MCP spec limitations."
   )
   ```

2. **自动包装基础设施** 增加了复杂性：
   - `_WrappedResult` dataclass 会包装非对象返回值
   - 自定义 `x-fastmcp-wrap-result` 扩展支持客户端侧解包
   - SDK 和客户端都需要匹配的包装/解包装逻辑

3. **真实 bug** 已经因这些变通方案而产生：
   - Issue #2455：没有 `type: object` 的 `$ref` schemas 导致服务器上的所有工具失效
   - Issue #2421：意外的 `{"result": ...}` 包装让用户感到困惑

这表明当前限制确实给生态系统带来了实际摩擦，而 SEP-2106 将消除这些问题。

### OpenAPI 先例

OpenAPI 规范也经历过类似的演进。OpenAPI 3.0 使用了 JSON Schema 的“扩展子集”，带有自定义限制（例如要求使用 `nullable: true`，而不是允许 `"null"` 作为类型）。

OpenAPI 3.1 做出了战略性决定，全面与 JSON Schema 2020-12 对齐，接受破坏性变更以消除摩擦。结果是：更好的工具兼容性和更少的生态系统混淆。

| OpenAPI 的问题                   | MCP 的对应问题                           |
| ----------------------------------- | ----------------------------------------- |
| `type` 必须是字符串，而不是数组    | `inputSchema` 只允许特定字段              |
| 不能使用标准的 null 处理            | schema 中不能使用 `oneOf`/`anyOf`         |
| 自定义 `nullable` 关键字            | 仅支持对象的 `structuredContent`          |
| 导致工具链混淆                      | 导致 SDK 变通方案                        |

MCP 可以从 OpenAPI 的经验中学习，而不是在几年内重复同样的演进过程。

## 向后兼容性

此变更在**线格式上向后兼容**，但具体表现取决于版本不匹配的方向。

### 兼容性矩阵

|                           | 新客户端（SEP 之后）                                        | 旧客户端（SEP 之前）                                                                                                              |
| ------------------------- | ------------------------------------------------------------ | --------------------------------------------------------------------------------------------------------------------------------- |
| **新服务器（SEP 之后）** | 完全兼容。                                            | **仅当服务器返回对象类型的 `structuredContent` 时**才兼容。`structuredContent` 中的数组/基本类型可能会出问题。 |
| **旧服务器（SEP 之前）**  | 完全兼容。现有的仅对象模式依然有效。 | 不变。                                                                                                                        |

这种不对称性在于：采用数组或基本类型 `structuredContent`（或在 `inputSchema` 中使用组合关键字）的新服务器，不能假定旧客户端会接受其响应。针对先前线格式编写的旧客户端，可能会拒绝不是 JSON 对象的 `structuredContent`，或者无法验证包含 `type`/`properties`/`required` 之外关键字的 `inputSchema`。

为了与旧客户端保持互操作性，**使用数组或基本类型 `structuredContent` 的服务器还必须输出一个包含序列化 JSON 的 `TextContent` 块**（这也与工具规范中的既有建议一致）。不理解非对象 `structuredContent` 的客户端可以回退使用文本内容。

### TypeScript / SDK 迁移

将 `structuredContent` 字段类型从 `{ [key: string]: unknown }` 扩宽为 `unknown`，对类型化使用方来说是一个**源代码破坏性变更**，尽管线格式没有变化。如下代码：

```typescript
const result = await client.callTool({ name: "get_weather", arguments: { ... } });
const temp = result.structuredContent?.temperature;        // 之前可以通过编译（类型为 unknown）
const city = result.structuredContent?.["city"] as string; // 之前可以通过编译
```

在变更后将不再通过类型检查，因为 TypeScript 不允许在未进行缩小判断的情况下对 `unknown` 进行属性访问：

```typescript
const sc = result.structuredContent;
if (sc && typeof sc === "object" && !Array.isArray(sc)) {
  const temp = (sc as Record<string, unknown>).temperature;
}
```

这种破坏是有意的——只要工具返回非对象，旧类型就并不准确——但 SDK 维护者**应该**：

- 在 SDK 发布说明中记录此次迁移。
- 在体验允许的情况下，提供类型化辅助工具（例如针对某个工具的 `outputSchema` 的泛型），以便使用方无需手写缩小判断。

### 迁移路径

- **服务器**：为了保持原有行为，无需迁移。若要使用数组或基本类型 `structuredContent`，也请同时输出序列化的 `TextContent` 作为回退。
- **客户端**：旧客户端在仅对象服务器上仍可正常工作。若要使用新增的灵活性，应接受 `structuredContent` 中的任意 JSON 值，并在存在 `outputSchema` 时据此验证。
- **SDK**：更新生成类型以反映新 schema（`structuredContent` 使用 `unknown`，`inputSchema`/`outputSchema` 为开放式），并在发布说明中指出这一源代码破坏性类型变更。

## 安全影响

JSON Schema 校验已经处理类型检查、值约束和必填字段验证，实现时**必须**继续根据声明的 schema 验证所有输入和输出。允许完整的 JSON Schema 2020-12 词汇集会带来两个需要明确说明的方面。

### `$ref` 解引用（SSRF 与 Fetch-DoS）

JSON Schema 2020-12 允许 `$ref` 指向绝对 URI，而不只是同一文档中的 JSON Pointer。一个天真的实现如果对遇到的每个 `$ref` 都发起 HTTP 请求来解析，就会给攻击者提供一个服务器端请求伪造 / 抓取放大原语：恶意工具定义可以导致宿主抓取任意 URL，包括内部元数据端点或旨在耗尽资源的大型负载。

为缓解此问题：

- 实现**不得**自动解引用解析到网络 URI 的 `$ref` 值（即任何不是同文档 JSON Pointer 的内容，例如 `#/$defs/Foo` 或内部 `$anchor`）。
- 这里的“自动”是指“作为正常验证或 schema 处理的一部分，而无需操作员显式操作”。实现**可以**提供一个可选开启的模式来抓取非本地 `$ref`，但默认**必须**关闭，并且**应该**强制执行主机白名单（或至少拒绝回环地址、链路本地地址和私有网络地址），应用超时和大小限制，并记录被解引用的 URI。
- 因未解析到外部 `$ref` 而验证失败的 schema，**应该**被拒绝，而不是悄悄当作宽松处理。

### 组合关键字的资源使用

组合关键字（`anyOf`、`oneOf`、`allOf`、`if`/`then`/`else`）和 `$defs` 使 schema 更具表达力，但某些病态组合的验证成本可能很高。实现**应该**施加合理的边界——例如最大 schema 深度、子 schema 总数上限，或每次验证的时间预算——以防止恶意工具定义成为针对验证器的 CPU DoS 向量。
