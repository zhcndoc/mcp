- **状态**: Final
- **类型**: Standards Track
- **创建时间**: 2026-03-11
- **作者**: Peter Alexander (@pja-ant)
- **赞助人**: Peter Alexander (@pja-ant)
- **PR**: https://github.com/modelcontextprotocol/modelcontextprotocol/pull/2567
- **相关**: [SEP-2575](https://github.com/modelcontextprotocol/modelcontextprotocol/pull/2575)（使 MCP 无状态），[SEP-2322](https://github.com/modelcontextprotocol/modelcontextprotocol/pull/2322)（多轮往返请求），[SEP-2549](https://github.com/modelcontextprotocol/modelcontextprotocol/pull/2549)（列表结果的 TTL）

## 摘要

本提案从 MCP 中移除协议层面的会话概念，用服务器签发的显式状态句柄替代隐式的会话范围状态，模型会携带这些句柄并在后续调用中传递它们。[SEP-2575] 移除了 `initialize` 握手，并按请求携带协议版本和能力；本提案是与之配套的变更，它移除了会话和 `Mcp-Session-Id` 标头。二者结合使 MCP 在协议层变为无状态。

在规范中存在一年多之后，会话在不同客户端之间并未收敛出一致含义：有些按工具调用划分，有些按应用启动划分，有些按页面加载划分，而几乎没有客户端会恢复会话。服务器作者无法预测其服务器连接到任意客户端时，会话的作用域或生命周期是什么，这使得会话作为应用状态容器变得不可靠。本提案认为，应用状态可以通过显式标识符来承载，而会话抽象增加了约束（固定基数、不明确的生命周期、跨会话边界的列表端点不可缓存），却没有相应收益。

在本提案下，当前如果某个服务器将购物车（例如）作用域限定到会话中，那么它将改为提供一个工具 `create_basket()`，返回一个 `basket_id`，并在后续工具调用中传递该 ID，例如 `add_item(basket_id, ...)`。模型决定哪些内容共享，哪些内容隔离；列表端点在曾经的会话边界之间变得可缓存；代理编排器可以根据需要自由共享或不共享应用状态。显式状态句柄不是一种新的协议构造——它们没有 schema 或线格式。它们是一种工具设计模式；协议层面的变化是移除会话，从而让句柄成为表达跨调用状态的方式。

## 动机

### 今天会话作用域涵盖什么

当前规范对于哪些行为属于会话绑定并不精确，但在实践中，有五类内容会附着于会话生命周期：

1. **协商出的能力和协议版本。** `initialize` 的结果——当前使用的是哪一版协议，以及双方支持哪些可选能力——是在每个会话中建立一次，并在该会话持续期间假定有效。[SEP-2575] 通过移除 `initialize` 并按请求携带版本/能力信息来解决这一点，因此本提案将其视为已被处理。

2. **提示获取（elicitation）和采样的中间状态。** 当工具调用触发 `elicitation/create` 或 `sampling/createMessage` 的往返时，服务器必须将最终响应与原始进行中的工具调用关联起来——而这些状态今天是隐式存在于会话中的。[SEP-2322]（多轮往返请求）通过在请求/响应循环中显式携带关联状态来解决这一点，因此本提案将其视为已被处理。

3. **应用状态。** 典型例子是购物车：`add_item()`、`add_item()`、`checkout()`，购物车按会话隐式存在。该模式可推广到任何有状态工作流——Playwright 浏览器实例、数据库事务、打开的文件描述符。

4. **可变列表端点。** `tools/list`（以及 `resources/list`、`prompts/list`）可以在会话生命周期内合法返回不同结果。例如，数据库服务器可以暴露一个 `connect_database` 工具，调用后会使 `query` 和 `list_tables` 出现在后续的 `tools/list` 结果中。

5. **资源订阅。** 订阅生命周期与会话生命周期绑定。（[SEP-2575] 引入了 `messages/listen` 作为服务器到客户端通知的投递通道；在该模型下订阅生命周期不在此重新审视。）

在 (1)、(2) 和 (5) 已由其他 SEP 处理的情况下，本提案处理 (3) 和 (4)。

### 会话作用域的问题

以下问题适用于会话是强制性的（当前规范）或可选的两种情况。

#### 会话生命周期未定义，服务器无法据此设计

规范没有说明会话何时开始或结束，因为这取决于宿主应用。实际上，已部署的客户端差异很大，而且很少将会话按对话划分：ChatGPT 会为每一次单独的工具调用创建一个全新的会话，Claude.ai 直到最近也是如此；[^per-call] 大多数桌面和 IDE 客户端会在应用启动时创建一个会话，并在进程生命周期内保持它；Web 客户端通常每次页面加载创建一个会话。几乎没有客户端会在断开连接或重启后恢复先前的会话，而在服务器端，参考 TypeScript SDK 并没有公开 API 可在不同节点上重建会话，因此即使客户端尝试恢复，多节点部署也无法支持这一点。[^^ts-sdk-resume] 子代理可能与其父代理共享会话，也可能拥有自己的会话——并无约定。

[^per-call]: [microsoft/playwright-mcp#1045](https://github.com/microsoft/playwright-mcp/issues/1045)，2025 年 9 月——服务器作者报告称 ChatGPT 和 Claude.ai 都会在每次工具调用后关闭会话，导致浏览器状态丢失；[“Connector tool calls generating fresh MCP session each invocation”](https://community.openai.com/t/connector-tool-calls-generating-fresh-mcp-session-each-invocation/1364975)，OpenAI 开发者社区，2025 年 11 月。

[^ts-sdk-resume]: [modelcontextprotocol/typescript-sdk#1658](https://github.com/modelcontextprotocol/typescript-sdk/issues/1658)，2026 年 3 月——`StreamableHTTPServerTransport` 将会话状态存储在私有实例字段中，没有从外部存储重新加载的 API。

这很重要，因为决定将什么内容作用域限定到会话的是服务器作者，而他们需要知道一个会话究竟对应什么，才能正确地这样做。将浏览器实例绑定到会话的 Playwright 服务器需要知道这意味着一次用户回合、一个代理进程，还是一个长生命周期的聊天。规范并未规定这一点，不同宿主给出的答案也不同，因此服务器是在对一个其语义无法控制的抽象进行设计。

实际后果是，会话作用域的应用状态往往无法保留。面对按工具调用的客户端，它会在下一次调用前被销毁；面对按应用启动的客户端，它会在窗口中的每一次对话中共享，然后在重启后丢失；面对任何不恢复会话的客户端，应用重启时它就消失了。看起来似乎在成功使用会话状态的服务器，通常是依赖进程生命周期的 stdio 服务器，而这属于传输层而非协议层的属性。

#### 列表端点不能跨会话缓存

由于 `tools/list` 可能依赖会话，客户端不能假定在一个会话中获取的结果在下一个会话中仍然有效。每个新会话都必须重新获取，即使服务器的工具集合在构建时就已固定且从不变化，这也是常见情况。

Python SDK 自己关于客户端侧列表缓存的设计问题，把“缓存键是什么——按会话还是按服务器 URL？”列为一个未决问题，[^^py-sdk-cache] 而网关实现者已经明确按会话实现缓存，因为他们无法依据规范假设跨会话有效。[^^agentgateway-cache] 所有列表端点都必须被视为可能按会话作用域，因此为了安全，每个会话都必须重新获取。

[^py-sdk-cache]: [modelcontextprotocol/python-sdk#2108](https://github.com/modelcontextprotocol/python-sdk/issues/2108)，2026 年 2 月。

[^agentgateway-cache]: [agentgateway/agentgateway#1510](https://github.com/agentgateway/agentgateway/issues/1510)，2026 年 4 月。

对于经常生成子代理的宿主来说，这会成为热点路径上的乘数效应。如果服务器可能按会话作用域，便会强制产生 `O(子代理数 × 服务器数)` 次 `tools/list` 调用：每个子代理、每个服务器、每次都要调用，即使底层工具集合自编排器首次连接以来没有变化。客户端不能跳过这次调用，因为它无法事先知道哪些服务器是按会话作用域的。对于生成许多短生命周期子代理的编排器来说，这一开销甚至可能超过实际工具调用的协议流量。在本提案下，同样的工作负载是 `O(服务器数)`：编排器为每个列表获取一次，之后所有子代理复用缓存结果。

如果列表端点只取决于服务器部署和已认证主体，客户端就可以缓存它们，并在收到显式信号时使其失效。[SEP-2549] 规定了这样的信号（服务器宣布的 TTL 加上 `notifications/*/list_changed`），但其缓存模型只有在列表不再按会话变化时才是可靠的。移除会话使这一模型安全；此时子代理可以零成本继承父代理的缓存列表。

#### 每个会话的基数固定为 1

会话状态在每个会话中的基数恰好为 1。模型只能得到一个购物车、一个浏览器、以及服务器作用域到会话中的任何一个对象；它不能有两个，也不能有零个。

当不同状态片段需要不同作用域时，这就成了问题。考虑一个编排器，它生成多个子代理来独立研究待购买的产品。子代理应该添加到同一个购物车中（它们是在协作完成同一笔订单），但每个子代理都需要自己的浏览器状态（它们在并行浏览不同的网站）。

没有任何会话边界能够同时满足这两点：

| 会话模型               | 购物车（期望：共享） | 浏览器（期望：隔离） |
| ---------------------- | :-----------------: | :------------------: |
| 子代理共享父会话       |      ✓ 共享        |   ✗ 共享（相互覆盖）  |
| 子代理拥有各自会话     |     ✗ 隔离          |        ✓ 隔离        |

使用显式 ID 时，编排器只调用一次 `create_basket()`，将得到的 `basket_id` 传给每个子代理，而每个子代理又分别调用 `create_browser()` 获取自己的 `browser_id`。模型可以针对每一份状态决定哪些应共享，哪些应隔离，而不是把一个作用域强加给所有内容。

缺少标识符的同样问题还意味着，会话状态无法从创建它的会话之外被引用。在一个聊天中创建的购物车对另一个聊天是不可见的；如果用户想在新的对话中恢复工作、把内容交接给另一个代理，或者与同事共享状态，会话模型都无法提供任何可指向它的引用。显式的 `basket_id` 则可以被传递给上述任何一种情况。

## 规范

### 变更摘要

1. **从协议中移除 session 概念。** `Mcp-Session-Id` 头部被移除，描述 session 生命周期和 session 作用域行为的规范语言也被删除。该协议在每一层都不再有 session。([SEP-2575] 移除了 `initialize` 握手，但明确将移除 session 的工作延后到本提案。)

2. **列表端点与 session 无关。** 在没有 session 的情况下，`tools/list`、`resources/list` 和 `prompts/list` 的结果不再有任何按 session 或按连接划分的作用域可依赖。列表仍然可能因其他原因而变化（服务器部署、认证变更）；这些情况的缓存和失效机制已在 [SEP-2549] 中单独规定。

3. **有状态工作流使用显式句柄。** 在 session 被移除后，需要在多次工具调用之间维持状态的服务器，可以通过在创建工具中返回一个标识符，并在后续调用中将其作为参数接收来实现。

第三点**不是**协议变更。不存在 `handles/*` 方法，schema 中也没有 handle 类型，wire 层面根本没有 handle 这个概念。从协议角度看，handle 只是工具结果中的一个字符串，以及工具参数中的一个字符串，与任何其他工具数据没有区别。“显式状态句柄”是一种工具设计模式，规范对其进行记录并推荐——就像它可能会记录分页或错误消息约定一样——而不是它所实现的内容。本 SEP 的规范性内容是第 (1) 点中的移除；(2) 点由此自然推出，而 (3) 点则是填补空白的指导。

### 显式状态句柄

#### 模式

在以前服务器会依赖隐式的 session 作用域状态——例如 `add_item` 调用操作的是每个 session 对应的购物车——现在它改为暴露一个显式的创建工具，并返回一个 handle：

```jsonc
// → tools/call
{ "name": "create_basket", "arguments": {} }

// ← result
{ "content": [{ "type": "text", "text": "Created basket bsk_a1b2c3" }],
  "structuredContent": { "basket_id": "bsk_a1b2c3" } }
```

然后模型将该 handle 作为普通参数传递到后续调用中：

```jsonc
// → tools/call
{ "name": "add_item",
  "arguments": { "basket_id": "bsk_a1b2c3", "sku": "shoes" } }

// ← result
{ "content": [{ "type": "text", "text": "Added shoes to bsk_a1b2c3 (1 item)" }] }

// → tools/call
{ "name": "checkout",
  "arguments": { "basket_id": "bsk_a1b2c3" } }
```

这里没有任何协议扩展：`basket_id` 只是 `structuredContent` 中的一个普通字符串字段，也是后续工具的普通字符串参数。这种模式已经是广泛部署的远程 MCP 服务器在管理持久资源时的常见做法：

| 服务器（官方，远程）                                   | 创建工具 → 返回的 ID         | 接收该 ID 的操作工具                                     |
| ----------------------------------------------------------- | --------------------------------- | ---------------------------------------------------------------- |
| [Linear](https://linear.app/docs/mcp)                       | `create_issue` → issue id         | `get_issue`, `update_issue`, `create_comment`                    |
| [Notion](https://developers.notion.com/docs/mcp)            | `notion-create-pages` → page id   | `notion-update-page`, `notion-move-pages`                        |
| [GitHub](https://github.com/github/github-mcp-server#tools) | `create_pull_request` → PR number | `pull_request_read`, `update_pull_request`, `merge_pull_request` |
| [Stripe](https://docs.stripe.com/mcp)                       | `create_customer` → customer id   | `create_invoice`, `list_subscriptions`                           |

对于生命周期较短的对象（浏览器上下文、正在进行中的购物车），也可以采用这种方法：为创建出的对象设定有限的生命周期，和/或将其可发现性限制为创建它的主体。状态由服务器持有，客户端持有它的名字，并且在每次调用时都会检查授权。

#### 服务器指南

以下内容均非规范性要求。句柄是一种工具设计模式，而不是协议特性；服务器可以根据自身领域自由塑造它们。该模式在以下情况下效果最佳：

- **句柄应当是不可解析的。** 如果句柄编码了内部结构（`cart_user42_2026-03-11`），就会诱使客户端解析它，或让模型去猜测它；而像 `bsk_a1b2c3` 这样的不可解析句柄则不会。
- **持有不等于授权（在存在认证的场景中）。** 对于已认证的服务器，每次调用都应验证 `(handle, auth_context)`；句柄最终会出现在聊天日志、复制粘贴缓冲区以及子代理提示词中。对于未认证的服务器，由于 handle 必然是 bearer token，应使用至少 128 位的加密安全熵来生成，并限制其生命周期。参见 [Security Implications](#security-implications)。
- **持久性应在工具描述中写明。** 句柄按设计会跨越连接存活，因此“状态持续到连接关闭为止”不再适用。把策略写进 `create_*` 工具的描述中——例如“返回一个 basket_id；basket 在空闲 24 小时后过期”——这样模型在决定是否创建状态时就能看到。仅写在服务器文档中的策略，模型是看不到的。
- **过期句柄应返回有用的错误。** 当工具收到一个指向已过期或已销毁状态的 handle 时，错误信息应明确说明——例如“basket `bsk_a1b2c3` 已过期”，而不是“无效参数”。清晰的过期错误可以让模型通过再次调用 `create_*` 来恢复；含糊的错误通常会导致重试或失败。
- **创建时接收参数。** `create_context(cluster="staging")` 优于先 `create_context()` 再 `set_cluster(ctx, "staging")`：这样只需一次往返而不是两次，而且状态不会以半配置状态存在。
- **应提供清理能力。** `destroy_*(handle)` 工具可以让模型释放资源。`list_*()` 工具可以让模型在丢失跟踪对象后恢复。两者都不是必需的。

#### 客户端指南

从客户端的角度看，handle 只是工具结果中的一个普通字符串。客户端的主要职责是确保这个字符串在上下文压缩后仍然保留；如果对话被总结，而 handle 位于被丢弃的部分，那么该状态就会变成孤儿状态。已经跨越压缩边界跟踪工具调用结果的客户端，现在本来就已经在处理这件事了。

### 与 session 无关的列表端点

在移除 session 后，列表端点不再有 session 可供变化依赖。本 SEP 对 `tools/list`、`resources/list` 和 `prompts/list` 施加的唯一约束就是：它们的结果不再有按 session 或按连接划分的作用域可依赖。这并不排除列表根据请求中携带的授权而变化：凭据会随每个请求一同携带，因此，如果服务器对不同主体或不同作用域返回不同的工具集，那依赖的是每个请求的输入，而不是连接状态。列表也仍然可能因其他原因随时间变化——服务器部署了新版本、用户的套餐或被授予的作用域发生变化——而本 SEP 不枚举也不限制这些情况。

客户端如何得知缓存列表已过期，是 [SEP-2549] 的主题。该提案定义了列表响应上的服务器声明 TTL，以及与 `notifications/*/list_changed` 的交互。这两个 SEP 是互补的：本提案移除了 session 作为变化来源，因此缓存对象变得稳定；[SEP-2549] 则规定了缓存多长时间以及何时失效。

上述约束的一个后果是，服务器不再能够将列表结果作为其他请求的副作用进行变更；在“动机”部分中的那种模式——调用 `connect_database()` 之后，`query` 和 `list_tables` 会出现在后续的 `tools/list` 结果中——不再被允许。为了达到相同效果，服务器在 list 时无条件暴露 `query` 和 `list_tables`，并让它们接收由 `connect_database()` 返回的 `connection_id` 参数。没有有效 `connection_id` 的 `query` 调用会失败，并返回一个错误，提示模型先调用 `connect_database()`；这种依赖关系是通过工具的输入 schema 和描述来表达的，而不是通过列表结果来表达。

### 由此产生的规范编辑

除了删除 §Session Management 小节本身之外，当前规范中的其他若干部分也使用 session 作用域来定义行为，因此需要重新定界：

- **JSON-RPC 请求 ID 的唯一性。** 当前规范要求请求 `id` “在同一 session 内，发起请求者之前未使用过”。`id` 的作用是让发送方将收到的响应与产生该响应的请求关联起来；接收方只需回显它。移除 session 后，相应约束也重新定界：发送方不得发出一个其 `id` 与自己已发送但尚未收到响应的另一请求相同的请求。这种约束与传输无关，足以在所有传输下进行关联，也是 TypeScript 和 Python SDK 通过每个客户端对象上的单调递增计数器所实现的方式。([JSON-RPC 2.0 §4](https://www.jsonrpc.org/specification#request_object) 本身并不施加唯一性要求——它只要求接收方回显 `id`——因此这仍然是 MCP 层面的约束。)
- **SSE 事件 ID 的唯一性。** 当前规范将 SSE 事件 ID 的作用域限定为“在该 session 内所有流中全局唯一”。移除 session 后，约束变为：该 ID 必须在服务器管理的所有流之间全局唯一，以便 `Last-Event-ID` 能解析到唯一的流。现有关于事件 ID 应编码其来源流的建议，实际上已经隐含了这一点。
- **分页游标有效性。** 当前规范建议客户端不要“跨 session 持久化游标”。移除 session 后，这条建议也随之消失。游标稳定性和快照一致性超出本提案的范围。
- **列表端点的可变性。** `tools/list`、`resources/list` 和 `prompts/list` 页面都写着结果“可能在连接生命周期内发生变化”。这些内容根据 [§与 session 无关的列表端点](#session-independent-list-endpoints) 重新定界：结果可以随时间变化，但不得按连接变化，或作为该连接上其他请求的副作用而变化。
- **措辞。** 少数使用“session”作描述的短语——架构概览中的“有状态 session 协议”、能力协商中的“在 session 期间可用”、授权中的“同一逻辑 session”、关于仅凭 session ID 关联状态的 elicitation 禁止条款，以及类似表述——都被改写或删除。除 session 移除本身之外，这些内容不带来任何语义变化。

## 理由

### 为什么要移除会话，而不是 պարզապես 默认关闭它们？

[SEP-2575] 已经通过移除 `initialize` 握手解决了让 MCP 在负载均衡器后面以及在没有粘性路由的情况下工作的问题。之所以还要进一步移除会话，而不是将其保留为可选能力，原因如下：

- **即使是可选会话，也仍然会阻止列表缓存。** 除非客户端知道服务器没有选择启用会话作用域变更，否则它无法在会话边界之间缓存 `tools/list`；而它无法提前知道这一点。因此，客户端会为每个服务器、每个会话重新获取，即使真正选择启用的服务器很少。动机部分中的 `O(subagents × servers)` 成本是由“会话可能存在”造成的，而不是由“会话实际被使用”造成的，所以把它做成可选并不能消除这个成本。
- **这一原语会影响服务器设计。** 在规范中提供会话作用域状态，会引导服务器作者将其用于本来更适合显式 ID 的工作流。
- **更少的原语意味着更小的实现面。** 每一个协议概念都必须由 SDK 作者实现、记录文档，并被新用户学习。

### 表达能力

一个会话在每个连接上恰好提供一个作用域。显式 ID 则提供与模型创建的对象数量一样多的作用域，并且每个作用域都可以独立共享或隔离。凡是会话能够表达的内容，都可以用模型在对话开始时创建的单个 ID 来表达；反过来则不成立。

### 恢复

因为句柄会出现在工具结果中，所以它们属于聊天记录的一部分。任何会持久化聊天内容的客户端——这也是大多数客户端——因此都会自动持久化这些句柄。在应用重启、页面刷新，或切换到另一台设备后重新打开对话时，句柄会再次出现在模型面前，无需额外的恢复机制，而且这种行为在各客户端之间是一致的。相比之下，基于会话的状态要求客户端在带外持久化并重新发送 `Mcp-Session-Id`，而正如动机部分所述，几乎没有客户端这样做。

### 预期的反对意见

#### 垃圾回收

会话提供了一个生命周期信号——当会话结束时，状态会被释放。没有它，模型可能会忘记调用 `destroy_basket()`，从而导致状态泄漏。

然而，在实践中，会话并不能可靠地提供这一点。正如动机部分所述，真实客户端要么从不结束会话（按应用启动周期），要么不断结束会话（按工具调用），要么在与对话无关的时刻结束会话（页面刷新、网络抖动）。位于负载均衡器后面的无状态 HTTP 服务器根本看不到连接关闭。服务器今天已经在依赖基于 TTL 的过期机制；会话边界并不是实际执行清理的东西。

带有文档化持久性策略的显式 ID（“basket 在 24 小时空闲后过期”）本质上是同一种机制，只是将其显式化了。

#### 模型必须把 ID 一路带下去

在隐式会话状态下，由服务器跟踪标识符；而在显式 ID 下，模型负责在每个相关调用中传递 `basket_abc123`。其失败模式要么是幻觉出一个略有错误的 ID，要么是在对话被压缩后 ID 从上下文中丢失。

模型已经在对话中经常可靠地传递不透明标识符——文件路径、URL、提交哈希、PR 编号、先前工具调用返回的 UUID——而当前模型在这方面做得相当可靠。压缩是更困难的情况，但它影响的是任何长程状态：如果压缩器丢掉了仍然有效的工具调用结果，那么模型也会失去对会话作用域购物车里有什么的跟踪，而不只是购物车的 ID。

#### 聊天历史中的 ID

一个可以被粘贴到任何地方的 `basket_id` 可能会变成用户聊天日志中的未经认证能力凭证。

对于已认证服务器，ID 应该只是一个名称，服务器应在每次调用时检查 `(id, auth_context)`。Google 文档 ID 会出现在 URL 和浏览器历史中；访问由 ACL 控制，而不是由 ID 保密性控制。这里也是同样的道理。

对于没有认证的服务器，ID 本质上就是一个 bearer token——服务器唯一能检查的就是是否持有它。在这种情况下，该句柄应遵循不可猜测能力令牌的标准做法：由密码学安全随机源生成，至少具有 128 位熵（例如 UUIDv4，或 22 个以上字符的 URL 安全 base64），绝不能由可预测输入派生，并且应设置有限生命周期。这与常见的其他一次性公开 ID 的做法相同——“知道链接即可访问”的分享 URL、密码重置令牌、Stripe Checkout session ID——并且带来相同的权衡：方便，但任何获得令牌的人在其生命周期内都拥有访问权限。

#### 破坏性变更

会话今天已经在规范中；移除它们会破坏任何依赖它们的实现。

对一个包含 1000 个仓库的开源 MCP 服务器随机样本进行了自动化调查（由每个仓库的 LLM 分析分类），结果发现：

| 类别                                                      | 占比  | 迁移方式                                             |
| ------------------------------------------------------------- | ----: | ---------------------------------------------------- |
| 没有对 MCP session ID 的应用层引用                             | 90.0% | 无                                                   |
| `Map<sessionId, Transport>` 路由（TS SDK 样板代码）            |  3.5% | 通过无会话 SDK transport 移除                       |
| 仅用于 transport 设置（`sessionIdGenerator`，从不读取）        |  2.8% | 删除一个构造器选项                                   |
| **以会话为键的应用状态**                                      |  2.5% | 迁移到显式句柄或认证主体                             |
| **代理 / 网关粘性路由**                                       |  0.7% | 需要专门设计替代方案                                 |
| **认证绑定**（JWT claims、以 session 为键的 PKCE verifier）   |  0.5% | 替换为服务器生成的 nonce 或 token subject           |

加粗的行是把 session ID 用于应用语义的仓库。受影响最严重的类别——每个会话启动一个上游的网关——需要专门设计的替代方案，而不是机械式修改；见[向后兼容性](#backward-compatibility)。

## 向后兼容性

这对依赖协议级会话状态的服务器来说是一个**破坏性变更**。迁移路径取决于服务器类别：

**使用进程生命周期状态的 Stdio 服务器。** 这类是当今最常见的有状态服务器。就默认部署而言，从机制上说它们不会被此提案破坏——进程生命周期仍然存在，而一个为每个进程保留单个内存中浏览器实例的服务器，继续与会为其启动一个进程的 stdio 客户端正常工作。不过，这类服务器**不应**依赖进程生命周期状态，而**应**迁移到显式句柄。进程生命周期存在与此 SEP 要为 HTTP 消除的同样的作用域未定义问题（一个进程对应一次对话、一次应用启动，还是别的什么，由宿主决定），而依赖它的服务器无法在 HTTP 上提供等价行为，因为 HTTP 下不存在每个客户端一个进程的概念。Stdio 服务器从来没有 `Mcp-Session-Id`，因此这个头部移除本身不会影响它们。

**使用 `Mcp-Session-Id` 的 HTTP 服务器。** 这类较少见，必须迁移到显式句柄。迁移是机械性的：将会话作用域状态映射替换为以句柄为键的状态映射，添加一个 `create_*` 工具，并将句柄作为参数添加到有状态工具中。

**将 session ID 用作遥测键的服务器。** 一些服务器会用 session ID 给 trace、日志或限流桶打标签，以便关联会话内的活动。这在不同客户端之间本来就表现不一致——对于按工具调用的客户端，每个事件都会落入自己的桶中；对于不会恢复的客户端，关联会在每次重启时断开。这些用例需要迁移到其他作用域机制，通常是认证主体（bearer token subject、API key）或请求级关联 ID。

**将 session ID 用于粘性路由的代理和网关。** 按 `Mcp-Session-Id` 路由的网关会失去其路由键——但它们之所以需要这个键，只是因为它们的上游是有状态的。如果上游是无状态的（或者迁移到显式句柄，其中状态键位于工具参数中，任何副本都可以从共享存储中提供服务），网关就根本不需要粘性路由。剩余的情况是那些通过为每个会话启动一个子进程来桥接 HTTP 到 stdio 的网关；它们需要一个不同的关联键，这是传输层的问题（按认证主体路由，或使用 cookie / 网关签发的头部），而不是此 SEP 所定义的内容。

**将认证工件绑定到 session ID 的服务器。** 少数服务器会把 OAuth PKCE verifier、session→user 绑定映射，或以 session ID 为键的 JWT claims 存储起来。在 PKCE 情况下，服务器已经通过 OAuth `state` 参数传递关联值（浏览器回调不是 MCP 请求，因此从未携带 `Mcp-Session-Id`），因此变更就是把服务器生成的 nonce 放进 `state`，而不是 session ID。session→user 绑定是为了应对[安全影响](#security-implications)中描述的 session 路由/认证解耦问题，而一旦每个请求都单独认证，这种绑定就不再需要。迁移大多是机械性的，不过由于涉及认证代码，仍值得审查。

**客户端。** 客户端会更简单：它们不再跟踪或重发会话标识符，也不需要判断某个服务器是否有状态。列表端点缓存将变得安全。

发布方式是一次干净切换：会话会在下一个规范版本中移除，没有弃用窗口。当前依赖会话作用域状态的服务器会继续停留在当前协议版本，直到它们迁移到显式句柄。协议版本协商已经处理了混合版本部署——支持两个版本的客户端会对尚未迁移的服务器使用旧协议，对其他所有服务器使用新协议。这样可以避免发布一个客户端同时支持两种模式的版本，因为那会阻止缓存收益（如果任何已连接服务器都可能是会话作用域的，客户端就不能缓存列表端点）。

## 安全影响

### 句柄暴露

此 SEP 引入的主要安全考虑是：句柄最终会出现在会话 ID 不会出现的地方——聊天日志、子代理提示、复制粘贴缓冲区，甚至可能出现在其他用户的屏幕上。

这改变的是暴露面，而不是引入一种新的漏洞类别。实际上，会话 ID 本身已经具有能力令牌的属性：例如，Python SDK 的有状态会话管理器仅通过 `Mcp-Session-Id` 路由，而不会验证请求中的已认证身份是否与创建该会话的身份一致，因此泄露的会话 ID 允许任何其他已认证主体劫持该会话。[^py-sdk-hijack] 下文“在每次调用时验证 `(id, auth_context)`”的建议同样适用于今天的会话 ID 和显式句柄；之所以在这个 SEP 中这一要求显得更明显，是因为句柄更显眼。

[^py-sdk-hijack]: [modelcontextprotocol/python-sdk#2100](https://github.com/modelcontextprotocol/python-sdk/issues/2100)。

对于已认证服务器，推荐的做法与 Google 文档 ID 和 GitHub PR 编号相同：ID 标识资源，而请求中的认证上下文决定访问权限。那些在每次调用时验证 `(handle, auth_context)` 的服务器不会受到句柄暴露的影响。

对于未认证服务器，没有可供检查的认证上下文，因此句柄就是一个能力令牌。这类令牌应至少使用 128 位的密码学安全熵生成，绝不能从可预测输入派生，并且应设置有界生命周期——这与“任何拥有链接的人”共享 URL 或密码重置令牌的做法相同。此类句柄一旦泄露，就会在其生命周期内授予访问权限；服务器应据此设定生命周期长度。

这只是指导建议，不是协议要求，因为协议中并不存在可供强制约束的句柄概念。

## 参考实现

除 PHP 之外，所有官方 SDK 都已经提供了无状态模式，其实现方式是不生成会话 ID（例如 TypeScript SDK 中的 `sessionIdGenerator: undefined`，Python SDK 中的 `stateless_http=True`）。这个 SEP 使该模式成为使用新协议版本的服务器的唯一选项。支持多个协议版本的 SDK 会保留为旧版本生成会话 ID 的代码路径；变化在于，当协商出的协议版本是此 SEP 引入的版本时，这条路径将不再可达。

## 未来工作

这个 SEP 有意不引入协议层面的句柄概念：从网络传输的角度看，`basket_id` 只是一个普通字符串。其结果是，客户端或模型并不会将 `basket_id` 标记为状态句柄——`create_basket` 的输出与 `add_item` 的输入之间的关系，是根据命名和工具描述推断出来的，而不是显式声明的。

后续提案可以将这种关系显式化，例如通过在服务器的工具输入和输出 schema 之间引用共享的 JSON Schema `$defs`，或者通过一种工具注解来标记某个结果字段为句柄。这样编排器就能识别哪些值是活动状态（用于压缩、交接或清理），而无需解析工具描述。这里不将其纳入范围，是为了让这个 SEP 保持为移除会话所需的最小内容。
